> 研究对象 = `[请填写工具/插件/AI工具的名称]`

你是一位擅长把复杂工具讲清楚的专家。请你对 `[工具名称]` 做一份**完全以用法为核心的深度解析**。不要写说明书，不要罗列功能。你的任务是让我在看完之后，能真正用它解决自己的问题，并且理解它设计上的巧妙与局限。

请按照下面五个大块来组织内容，但不要把它们当作僵硬的章节标题——你可以用更自然的叙述方式过渡。

### 第一块：这玩意儿是干嘛的？我什么时候会想起它？

- **一句话说清**：用最通俗的话定义这个工具的核心价值（像跟外行介绍一样）。
- **它消灭了什么麻烦**：在它出现之前，人们要做同样的事有多痛苦？举一个真实、具体的例子。
- **它最适合的三个场景**：每个场景用一句话说明「谁在什么情况下会用它」。
- **它绝对不适合做什么**：防止我硬用它做不擅长的事（这点很重要）。

---

### 第二块：最小行动闭环 —— 15 分钟内获得正反馈

- **最简安装/激活方式**（如果无需安装就跳过）。
- **第一个成功操作**：喂给它最简单的输入/指令，让它吐出正确的结果。把步骤写清楚，附带「如果你看到 xxx 提示，说明成功了」的验证方法。
- **接下来三个最高频的操作**：不是罗列快捷键，而是解释「当我想做 A 时，我通常会按什么 / 点哪里 / 输入什么」。

---

### 第三块：从零到够用 —— 用真实任务串起功能

请选取 **3 个最典型的真实任务**（难度递进），把完成每一个任务的完整思考+操作过程写出来。

每个任务按这个结构写：

1. **任务描述**：一个具体、有画面感的场景（例如「我需要从一篇长文章里提取所有含年份的句子」）。
2. **不用它的话，我本来怎么做**（一句话点出痛点）。
3. **用这个工具的步骤**：
  - 每一步写清楚「做什么」和「为什么这么做」。
  - 如果某步有多个分支（例如不同平台/不同输入类型），只讲最常用那条，再补充一句「如果你遇到…情况，可以改成…」。
4. **预期结果**：正确输出长什么样？
5. **常见卡点**：新手最容易在这一步犯什么错？报什么错？怎么救？

三个任务最好覆盖：

- 一个纯基础操作（打开 → 处理 → 保存）
- 一个稍微带点配置或参数调整
- 一个需要结合其他工具/插件完成

### 第四块：它跟其他工具怎么选？怎么一起用？

不要写成竞品参数表。请用对话式的对比：

- **如果你已经会 X 工具，什么时候该改用这个？**（例如：你会用 Excel 筛选，什么时候该用 Python pandas？）
- **如果你只会这个工具，什么时候该去学另一个？**（诚实说出它的边界）
- **它和哪些工具是黄金搭档？** 举一个组合使用的例子（比如：VS Code + 某个插件；或者 ChatGPT + 某个浏览器扩展）。

### 第五块：暗坑、谣言与进阶信号

- **最常被骂的三个反人类设计**（如果有），以及老手是怎么绕过或习惯的。
- **一个广泛流传但其实是错的用法**（例如「必须每次手动清理缓存」其实不需要）。
- **怎么知道自己已经「会用」了？** 给三个信号（例如：不再需要查某条命令的语法；遇到报错能猜出问题方向）。
- **想继续深入，下一步看什么？** 给 2-3 个最靠谱的资源（官方文档哪一章、哪个社区、哪个必读教程）。

### **【核心约束】真实性规范 —— 必须严格遵守**

为了减少幻觉、确保你给我的信息可以信赖，请你遵守以下规则。**任何一条违反，都会严重降低这份指南的价值。**

1. **区分「事实」与「推测/经验」**
  - 凡是来自官方文档、官方博客、权威源码注释、确凿的可查证来源，可以直接陈述。  
  - 如果是你自己的使用经验、推理、或者常见社区说法但未获官方确认，**必须明确标注**，例如：  
    > 「根据我的使用经验…」「社区中常见的一种做法是…」「官方文档未明确说明，但我实测发现…」「推测是因为…」  
  - 严禁将推测伪装成事实。
2. **参数、命令、快捷键、界面路径必须可验证**
  - 给出具体的命令或代码时，如果该工具版本不同有差异，请注明你基于的版本（例如 `v2.4.0`）。  
  - 不要编造不存在的参数或菜单项。如果某个功能你记不清了，宁可说「我不确定，请查阅官方文档某处」，也不要瞎写。  
  - 对于重要的命令，建议附带简短解释每个参数的作用，以证明你不是随意拼凑。
3. **尽可能引用来源**
  - 当提到某个功能的设计原因、某个限制、某个已知 Bug 或最佳实践时，最好给出可追溯的来源（如「官方文档《XXX》章节提到…」「GitHub issue #xxxx 中的开发者回复…」「2024 年 3 月的版本更新日志…」）。  
  - 如果信息来源是「网上很多人说」但无法定位到可靠出处，请如实说「这是一个常见的社区说法，但我未找到官方确认」。
4. **不要编造示例数据或错误信息**
  - 示例中的文件名、报错信息、输出内容，要么来自真实运行结果，要么明确说明「这是一个示例，实际输出可能因版本而异」。  
  - 不要用假造的报错信息来吓唬用户，除非那是官方文档中列举的典型错误。
5. **承认不确定性**
  - 如果某一部分你自己也不确定，直接说「抱歉，关于这一点我没有足够可靠的信息，建议查阅官方资源：…」。**这比硬给一个错误答案好得多。**
6. **鼓励「以官方文档为准」的提示**
  - 在指南的合适位置（如最后），可以加一句：  
    > 「本指南基于截至 YYYY-MM-DD 的公开信息。由于工具快速迭代，如有疑问请以官方最新文档为准：`[官方链接]`。」

### 写作风格要求（非常重要）

- **像对朋友说话**：用「你」「我」「我们」，不用「用户」「学习者」。
- **用故事和场景代替抽象描述**：不要写「支持批量处理」，要写「比如你有一百张图片要改尺寸，只需要敲一行…」。
- **每一段都要有信息量**：不说「这个功能很强大」这种空话，直接说「它能让你在 3 秒内完成以前需要 10 步才能做的事」。
- **不怕重复，怕遗漏**：关键步骤可以换个角度再说一遍。
- **诚实 > 完美**：宁可少写一个不确定的功能，也不要写出错的内容。
