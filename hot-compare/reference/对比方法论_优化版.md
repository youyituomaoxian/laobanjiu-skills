# GitHub 项目对比方法论（优化版）

> 基于原 github对比.md 四步法，补全 5 个漏洞后形成优化版 5 模块结构。
> 设计目标：让阅读者不仅"了解差异"，更能"学会使用"、"学会组合"，成为高阶玩家。

---

## 全局约束

1. **溯源优先**：所有功能/用法/流程以项目 README.md 和官方文档为唯一依据。无记载处标注「文档未记载，暂无可靠信息」。严禁编造。
2. **术语通俗化**：技术术语附带通俗解释，兼顾非技术读者。
3. **拒绝笼统**：功能、步骤、场景描述尽量具体，给真实命令和代码。
4. **对比对象≥3**：表格和流程图保持清晰，维度不缺失。
5. **长度限制**：全文 8000-15000 字。对象越多、差异越大，篇幅越长。

---

## 模块一：单项独立深解（7 项，逐个对象分析）

### 1.1 基础档案
- 项目/Skill 简介（一句话定义）
- 维护方/开发者
- 社区热度（Stars / Forks / Contributors）
- 最后更新时间 / 最新版本
- 开源协议（GitHub 项目必填）
- 生态活跃度（Issue 响应速度、PR 合并频率）

### 1.2 核心功能
- 主打能力清单
- 特色功能（独有、竞品没有的）
- 功能边界（明确能做什么、不能做什么）
- 反场景（绝对不适合的用法）

### 1.3 ⭐ 最简上手路径（新增）
- **一行安装命令**
- **第一个操作验证**：「输入 X，看到 Y 则成功」
- **老手才知道的捷径**：2-3 条实用技巧

### 1.4 使用方式
- 接入/调用/部署流程（步骤化）
- 操作步骤（从安装到产出）
- 交互形式（CLI / API / GUI / Skill 自动激活）
- 配置项复杂度评估

### 1.5 环境与依赖
- 运行系统要求
- 软件依赖清单
- 硬件/算力要求
- 网络要求

### 1.6 使用成本
- 免费 / 付费模式
- 有无付费版本及差异
- 资源占用（内存/算力/存储）

### 1.7 适用场景与人群
- 主流应用场景（3 个最典型）
- 目标使用者画像
- 最佳使用环境

### 信息缺失处理
当上述任何一项在 README 中无法找到时，统一标注：
> 「文档未记载，暂无可靠信息。建议查阅：[可能的文档链接]」

## 模块四：组合串联分析

### 4.1 原生兼容性总判

逐一检查各项目 README 中是否有：
- 官方推荐的集成/搭配方案
- API 对接文档
- Plugin/Ecosystem 支持

### 4.2 原生组合方案（如适用）

若有官方推荐组合：
- 组合名称和说明
- 完整使用流程（分步骤）
- 每步分工明细（哪个工具干什么、输出什么给下一个）
- 配置要点和避坑

### 4.3 ⭐ 非兼容串联方案（新增，关键补充）

**即使两个工具无法原生对接，也可以在一个实际工作流中串联使用。**

要求：
- 不编造 API 或集成方式
- 基于各工具自身的能力边界，设计「做完 A → 输出结果 → 手工/脚本传递给 B」的实际路径
- 标注每一步的人工介入点和自动化可行性

示例结构：
```
工作流：完成「XXX」任务
  Step 1：用 A 做 _____ → 产出文件/data
    A 负责：_____
  Step 2：手工/脚本将产出传递给 B → B 做 _____ → 最终产出
    B 负责：_____
  全流程耗时：~X min
  可自动化的步骤：Step 1 和 2（用 shell 脚本串联）
  必须人工的步骤：无
```

若确实无法串联（功能完全正交、无任何输入输出对接点），写明原因。
